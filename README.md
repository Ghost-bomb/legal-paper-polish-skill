# Reader-First Legal Writing

读者疑问驱动的中文法学论文写作教练。一个用于 TRAE（或其他 AI 编程助手）的 Skill，帮助法学研究者从"读者视角"诊断和优化论文段落的论证表达。

## 这个 Skill 不是什么

- **不是**自动 C 刊化润色器——不会一键生成"终稿"
- **不是**整篇重写工具——默认逐段处理，不跳过作者的意图说明
- **不是**降 AI 率工具——不承诺降低 AIGC 检测分数
- **不是**代写工具——不替作者补充观点、案例、文献或脚注

## 这个 Skill 是什么

一个**表达训练与读者视角诊断**的写作教练。核心思路是：先让作者用大白话说清楚"这段想让读者理解什么"，再回头检查原文是否真的把这个意图自然地传达了出来。

## 适用场景

- 论文段落读起来生硬、机械、像 AI 写的，想找到问题根源
- 想把知识点堆砌转变为读者疑问驱动的论证推进
- 想识别段落功能、读者疑问、过渡衔接和材料使用是否合理
- 需要对比"原文"和"作者真正想表达的意思"之间的差距
- 需要学术化表达参考，但不改变法律实质内容

## 工作流程

1. **确认范围**——用户指定要处理的段落
2. **切分段落**——按自然段拆分，标注每段的初步功能（背景、张力、概念界定、主张、材料引入、规则分析、回应质疑、限定、过渡等）
3. **大白话说明**——引导用户用大白话解释每段想让读者理解什么，不跳过这一步
4. **检查大白话**——如果大白话说得含糊、自相矛盾或与原文不一致，先追问而非直接改写
5. **诊断原文**——识别中心判断、读者疑问、材料功能、不自然句子、AI 法学论文腔、过度抽象表达、缺失的过渡
6. **学术表达参考**——给出"稳妥贴近原文版"和"自然推进增强版"两种参考，不声称是唯一终稿
7. **安全提示**——标注需要人工复核的脚注、引文、规范、案例、数据、术语、归因、条件、例外、因果关系和判断强度

## 与普通润色器的区别

| 普通润色器 | 本 Skill |
|---|---|
| 优化句子 | 诊断读者为什么卡住 |
| 添加抽象过渡词和正式连接词 | 去掉不必要的包装，检查每个判断是否因读者需要而出现 |
| 可能用 AI 理解替代作者意图 | 保留作者的框架、论点、脚注支撑和论证方向，改写前需作者确认真实意图 |

## 使用限制

- 不得复制、仿写或迁移语料库中的具体观点、材料、案例、脚注、文献或具有识别度的原句
- 不承诺 C 刊发表质量
- 不改变脚注所支撑的命题
- 不添加未经确认的实质性观点、案例、规范或文献
- 不过度使用"问题在于/进一步而言/由此可见"等模板句式
- 不用抽象名词堆砌来显得成熟
